## 1. 目标 (Goals)

建立一套**全自动每日微信群信息汇总系统**，解决以下两个问题：

1. **每日去重合并（主 — 80%）**：覆盖多个微信群，避免每天手动翻群、避免跨群转发被重复消费；每天早 8:00 向用户的 Telegram bot 推送精简版当日摘要。
2. **长期机会沉淀（次 — 20%）**：从日流水里结构化提取有长期价值的信息（邀请码、银行产品、优惠活动），构建可搜索的个人机会库；短期套利信息单独走"热点板"滚动淘汰机制，避免污染长期库。

## 2. 非目标 (Non-Goals)

- **不支持多用户**。纯单人工具，不做账号体系、不做权限。
- **不做实时推送**。每天一次批处理足够，不追求即时性。
- **不爬取/监控其他人的微信数据**。只读本机当前登录账号的本地 WeChat 数据库。
- **不做通用的聊天分析平台**。专门服务当前用户的薅羊毛/理财/套利信息场景。
- **不支持 Windows / Linux**。只支持 macOS（因为依赖 wx-cli 的 macOS ad-hoc codesign 流程）。
- **不提供 Web UI**。所有交互通过 TG 推送 + 本地 md 文件。

### 关键决策

- **全本地**：所有数据不出本机（除最终推送到 TG 的精简摘要）。
- **CLIProxyAPI 替代付费 API**：使用用户现有 Claude 订阅的 OAuth 凭证，通过本地代理暴露 OpenAI/Claude 兼容端点，零 API Key 成本。
- **LLM 做合并+总结，规则做抽取**：纯文本合并交给 LLM（能理解语义），硬指纹（URL/邀请码/MD5）用正则抽取保证确定性。

## 4. 数据模型 (Data Model)

### 4.1 permanent.jsonl — 永久机会库

**Append-only 的一行一个 JSON 对象**。适用场景：邀请码、银行产品、有明确截止日期的活动。

```json
{
  "id": "2026-04-17-bitget-invite",
  "captured_at": "2026-04-17T13:56:00+08:00",
  "source_group": "贝利知识星球VIP群❤️",
  "source_sender": "贝利Baily",
  "category": "invite_code",
  "type": "permanent",
  "title": "Bitget 邀请码",
  "content": "PjGhDx",
  "url": null,
  "expires_at": null,
  "last_mentioned_at": "2026-04-17T13:56:00+08:00",
  "mention_count": 1,
  "status": "alive",
  "death_signal": null,
  "notes": "护照可注册"
}
```

**字段说明：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 唯一 ID，格式 `YYYY-MM-DD-<slug>` |
| `captured_at` | ISO8601 | 首次抓到的时间 |
| `source_group` | string | 来源群名 |
| `source_sender` | string | 首次分享人昵称 |
| `category` | enum | `invite_code` / `bank_product` / `activity` / `misc` |
| `type` | enum | `permanent` / `product` / `activity` |
| `title` | string | 简短标题 |
| `content` | string | 主要信息（邀请码本身、链接、详情） |
| `url` | string? | 相关链接 |
| `expires_at` | ISO8601? | 明确截止时间（如有） |
| `last_mentioned_at` | ISO8601 | 最后一次在群里被提到的时间（后续有重复出现会更新） |
| `mention_count` | int | 累计被提及次数 |
| `status` | enum | `alive` / `likely_dead` / `dead` / `unknown` |
| `death_signal` | string? | 死亡信号原文摘录（被标 dead 时填） |
| `notes` | string? | 补充说明 |

### 4.2 permanent.md — 自动生成的人读视图

从 `permanent.jsonl` 自动生成，按 `category` 分组、按 `captured_at` 降序。用户不手动编辑。每次 pipeline 跑完自动 regenerate。

### 4.3 hot-leads/YYYY/MM/DD.md — 热点板日文件

**只在当日有新增热点时创建**（空日跳过）。每文件结构：

```markdown
# 2026-04-17 热点板新增

## OpenAI Plus 低价卡密（套利窗口）
- 来源：贝利知识星球VIP群 (贝利Baily 18:13)
- 内容：86GameStore / ferrigpt.asia，plus 6 元 / 5x 95 元 / 20x 160 元
- 本质：Apple Pay 英区/土区收据校验 bug
- 风险：建议小号，官方可能随时堵
- 状态：alive
- 首次抓取：2026-04-17
- 淘汰日期：2026-05-01（14 天后）
```

### 4.4 hot-leads/latest.md — 活跃聚合视图

每日 regenerate，聚合过去 14 天内 `status = alive` 的所有热点板条目。淘汰规则：

- 14 天到期自动标 `likely_dead` 并从 `latest.md` 移除（历史日文件保留）
- 被死亡信号命中直接标 `dead` 并立即移除
- 用户想手动续期：在 hot-leads/latest.md 里手动加注释，next pipeline 会尊重（TODO）

### 4.5 archive/YYYY/MM/DD/

每日原始物料 + 详细总结：

- `<群名>.md`：wx-cli 原始导出
- `summary.md`：LLM 生成的详细分析（完整的"每个群总结 + 值得关注清单 + 人物画像"）

## 5. 工作流 (Pipeline Details)

### 5.1 导出阶段

```bash
for group in $(yq '.groups[]' config.yaml); do
  wx export "$group" --since $YESTERDAY --until $TODAY \
    --format markdown -o archive/$YEAR/$MONTH/$DAY/"$group".md
done
```

### 5.2 指纹提取

对每条消息文本用正则抽取：

- URL：`https?://[^\s]+`（过滤表情/图片 CDN）
- 邀请码：粗启发式（5-12 位字母数字混合，且在"邀请码/推荐码/invite"上下文里）
- 图片/表情 MD5：XML content 里的 `emoticonmd5="..."`
- 文件 MD5：XML 里的 `cdnthumbmd5` / `md5`
- 手机号：`1[3-9]\d{9}`（不做脱敏，仅作为去重指纹）

### 5.3 跨群去重

指纹相同的消息视为同一条事件：

- 合并为一条，保留最早时间，`groups` 字段列出所有出现过的群
- **长文本**（>30 字符）走 SHA256 内容哈希做精确匹配
- **短文本**（≤30 字符）不去重（噪音太大，会把"好的"/"bd"/"[表情]"等高频短回复误当成重复）

### 5.4 LLM 总结

**Prompt 结构（给 CLIProxyAPI）：**

```
你是一个薅羊毛/理财信息分析助手。以下是用户昨天 N 个微信群的完整聊天记录（已去重）。

请输出两份 markdown：

### 精简版（≤1500 字，给 Telegram 推送）
1. 今日概览（2-3 句话总体评价）
2. 值得关注清单（表格：类型/内容/出处）
3. 详细档案路径（附在末尾）

### 详细版
1. 每个群 2-3 句话总结
2. 当日主干脉络
3. 值得关注清单（完整）
4. 人物画像（主要贡献者）

同时请以 JSON 格式输出"机会抽取"：
- 哪些条目应进永久库（永久型/产品级/活动级 + 明确截止）
- 哪些条目应进热点板（套利/漏洞/个人操作/灰产路子）
- 扫描是否有死亡信号命中某条已知机会（给出死亡信号原文）

{上下文：当前永久库活跃条目摘要}
{上下文：热点板 latest.md 内容}
{原料：去重后的今日消息}
```

### 5.5 死亡信号扫描

LLM 在 prompt 里拿到**当前活跃的 permanent/hot-leads 条目摘要**，扫描当日消息中是否出现"关门了/封了/不行了/过期了/没了/跑路了/开不了了"等关键词，并要求 LLM 判断该死亡信号是否明确指向某条 ID。

LLM 返回的 JSON：

```json
{
  "death_signals": [
    {
      "target_id": "2026-04-17-chase-vx-rebate",
      "signal_text": "路子早关门了",
      "signal_source": "默默ㄇㄛ 贝利VIP 21:32",
      "confidence": "high"
    }
  ]
}
```

`confidence=high` 自动标 `dead`；`medium` 标 `likely_dead` 需用户手动确认；`low` 忽略。

### 5.6 TG 推送

- 调 TG Bot API 的 `sendMessage` 接口，chat_id = 用户自己的 ID
- Markdown_v2 格式
- 消息末尾附 `详情：~/wx-daily/archive/YYYY/MM/DD/summary.md`
- 如果精简版 >4096 字符，拆分为多条连续消息

### 5.7 失败处理

- 每个 HTTP 调用（CLIProxyAPI / TG API）失败后 exponential backoff 重试 3 次
- 3 次全失败：调用 `osascript -e 'display notification ...'` 发 macOS 原生通知
- 所有失败写入 `~/wx-daily/logs/YYYY-MM-DD.log`
- 下次 Claude Code 启动时用户可在对话里问 "昨天 pipeline 有问题吗"，检查日志

## 7. 配置 (Configuration)

### 7.1 `~/wx-daily/config.yaml`

```yaml
# 要监控的微信群列表（名称必须与 wx sessions 输出一致）
groups:
  - "贝利知识星球VIP群❤️"
  - "贝利知识星球VIP2️⃣群❤️"
  - "OpenCLI 交流群"

# 打算加但还没加的群（TODO 区 — 提醒自己）
todo:
  - "<TODO: 其他值得监控的群>"

  time: "08:00"                       # 每天推送时间
  coverage: "yesterday"               # 覆盖昨天 00:00-24:00
  timezone: "Asia/Shanghai"

# 热点板滚动天数
hot_leads:
  retention_days: 14

# LLM 配置
# ────────────────────────────────────────────────────────────────────
# 📍 升级模型时改这里的 `model` 字段即可（无需改代码、无需改 endpoint/api-key）
# 当前选型：claude-sonnet-4-6（antigravity OAuth 池 / 走 Claude Code 订阅，不额外花钱）
# 备选升级：kimi-k2.5（Moonshot 中文特化，需中文语义把握更细时切换）
# 未来有更好的模型出现时，用 `curl http://127.0.0.1:8317/v1/models` 查当前可用列表
# ────────────────────────────────────────────────────────────────────
llm:
  endpoint: "http://127.0.0.1:8317/v1"     # CLIProxyAPI 本地端点（实测）
  model: "claude-sonnet-4-6"               # ← 升级时改这行
  api_key_env: "CLIPROXY_API_KEY"          # 从环境变量读，不硬编码
  max_tokens: 8000

# Telegram
telegram:
  bot_token_env: "TG_BOT_TOKEN"       # 从环境变量读 token（.zshenv 或 launchd plist）
  chat_id_env: "TG_CHAT_ID"           # 自己的 chat_id

# 失败处理
retry:
  max_attempts: 3
  backoff_seconds: [5, 15, 60]

# 敏感内容过滤
sanitize:
  enabled: false                      # d1: 不脱敏
```

### 7.2 launchd plist

`~/Library/LaunchAgents/com.apple.wx-daily-tg.plist`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.apple.wx-daily-tg</string>
    <key>ProgramArguments</key>
    <array>
        <string>/Users/Apple/projects/wx-daily-tg/venv/bin/python</string>
        <string>/Users/Apple/projects/wx-daily-tg/run_daily.py</string>
    </array>
    <key>StartCalendarInterval</key>
    <dict>
        <key>Hour</key>
        <integer>8</integer>
        <key>Minute</key>
        <integer>0</integer>
    </dict>
    <key>RunAtLoad</key>
    <false/>
    <key>StandardOutPath</key>
    <string>/Users/Apple/wx-daily/logs/stdout.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/Apple/wx-daily/logs/stderr.log</string>
    <key>EnvironmentVariables</key>
    <dict>
        <key>PATH</key>
        <string>/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin</string>
    </dict>
</dict>
</plist>
```

**关于补跑机制**：`RunAtLoad=false` 避免每次登录都跑。launchd 的 `StartCalendarInterval` 本身带有"**missed interval catch-up**"特性——若 Mac 在 8:00 处于睡眠/关机状态，唤醒/开机后 launchd 会在短时间内触发一次。这是系统默认行为，无需额外配置。

## 9. 错误处理 (Error Handling)

| 场景 | 行为 |
|---|---|
| Mac 8:00 睡眠 | launchd 醒来后自动补跑（delay 内补） |
| WeChat 未启动 | wx-cli 会失败，pipeline 记录错误，TG 推送"今日数据不可用，请启动微信"|
| CLIProxyAPI 未启动 | HTTP 连接失败，重试 3 次后 macOS 通知 |
| TG API 网络问题 | 重试 3 次，失败后 macOS 通知，精简版存 `logs/failed_pushes/` 待下次重发 |
| 某个群名改了 | wx-cli 找不到该群，单个群跳过，其他群继续，记录 warning |
| JSON 解析失败 | LLM 输出解析失败则降级为纯文本总结，机会抽取跳过 |
